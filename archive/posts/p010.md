## 错误处理：绕不过的现实

每写一行代码都要面对"如果失败了怎么办"：

- 打开文件——文件不存在
- 读用户输入——输入了字母而不是数字
- 申请内存——内存不够了
- 数组下标——越界了

不写错误处理的程序在演示时永远跑得很顺——但只要被真用户用，五分钟就崩。**写真实程序，错误处理的代码量经常比"主逻辑"还多**。

C++ 给了两套机制：

- **异常（exception）**：抛出/捕获，跳出正常控制流。
- **错误码 / `std::optional` / `std::expected`**：用返回值带回错误信息。

不同场合各有适合的选择。这篇先讲异常，再讲什么时候**不用**异常。

## 异常的三件套：throw / try / catch

### 一个最小例子

```cpp
#include <iostream>
#include <stdexcept>

double divide(int a, int b) {
    if (b == 0) {
        throw std::runtime_error("除数不能为 0");
    }
    return double(a) / b;
}

int main() {
    try {
        std::cout << divide(10, 2) << "\n";   // 5
        std::cout << divide(10, 0) << "\n";   // 这里会抛
        std::cout << "这行不会执行\n";
    } catch (const std::runtime_error& e) {
        std::cout << "出错了：" << e.what() << "\n";
    }
    std::cout << "main 继续往下走\n";
}
```

输出：

```
5
出错了：除数不能为 0
main 继续往下走
```

发生了什么：

1. `throw` **抛出一个异常对象**——通常是 `std::runtime_error` 这种从 `std::exception` 派生的类。
2. 抛出后，**正常控制流被打断**——`divide` 立刻返回，`main` 里 `divide(10, 0)` 之后那一行也不执行。
3. 一路往上找，直到遇到一个能匹配类型的 `catch`。
4. 进入 `catch` 块处理错误，`main` 后面的代码继续。

### `try` / `catch` 的形状

```cpp
try {
    // 可能抛异常的代码
} catch (const SomeException& e) {
    // 处理 SomeException
} catch (const OtherException& e) {
    // 处理 OtherException
} catch (...) {
    // 兜底：什么都接
}
```

**几个细节：**

- 多个 `catch` **从上到下依次匹配**，匹配到第一个就停。
- 永远写 `catch (const E& e)`——**const 引用**，不是按值。按值会拷贝异常对象，按引用又能用多态。
- `catch (...)` 是兜底，类似 `default`。能不写就不写——你都不知道接到了什么，处理起来很尴尬。

### `throw` 重新抛出

`catch` 里能"看一眼但不处理"，再扔出去给上层：

```cpp
try {
    do_something();
} catch (const std::exception& e) {
    log("got error: " + std::string(e.what()));
    throw;          // 重新抛出**当前异常**
}
```

注意是空的 `throw;`——抛出"刚抓到的那个"。`throw e;` 也合法但**会切片**（slicing）丢掉派生类信息。

## 标准异常类

`<stdexcept>` 里有一组现成的异常类，全部派生自 `std::exception`：

```
std::exception
├── std::runtime_error          // 运行时才能发现的错
│   ├── std::range_error
│   ├── std::overflow_error
│   └── std::underflow_error
├── std::logic_error             // 程序逻辑错（理论上能在写代码时避免）
│   ├── std::invalid_argument
│   ├── std::domain_error
│   ├── std::length_error
│   └── std::out_of_range
└── std::bad_alloc               // new 失败
└── std::bad_cast
└── ...
```

**入门阶段最常用的：**

- `std::runtime_error`：运行时错，参数是错误信息。
- `std::invalid_argument`：参数不合法。
- `std::out_of_range`：下标越界。

`std::vector::at()`、`std::stoi()` 这些标准库接口失败时抛的就是上面这几种。

```cpp
#include <vector>
#include <stdexcept>

std::vector<int> v{1, 2, 3};
try {
    std::cout << v.at(10);                  // 抛 std::out_of_range
} catch (const std::out_of_range& e) {
    std::cout << "越界：" << e.what() << "\n";
}

try {
    int n = std::stoi("not a number");      // 抛 std::invalid_argument
} catch (const std::invalid_argument& e) {
    std::cout << "字符串不是数字：" << e.what() << "\n";
}
```

## 自定义异常

继承 `std::runtime_error` 或 `std::exception` 就行：

```cpp
class FileNotFoundError : public std::runtime_error {
public:
    explicit FileNotFoundError(const std::string& path)
        : std::runtime_error("file not found: " + path), path_(path) {}

    const std::string& path() const { return path_; }

private:
    std::string path_;
};

try {
    throw FileNotFoundError("/etc/missing.conf");
} catch (const FileNotFoundError& e) {
    std::cout << e.what() << "\n";
    std::cout << "出错的文件是：" << e.path() << "\n";
}
```

::: callout 自定义异常的两个原则
1. **从 `std::exception` 体系派生**——这样别人写 `catch (const std::exception&)` 也能接到你。
2. **带上能帮诊断的信息**：路径、参数、错误码……异常本身就该是个数据载体。
:::

## RAII 在异常里的关键作用

异常抛出时**正常控制流被打断**，没执行到的 `delete`、`fclose`、`unlock` 都漏了——内存泄漏、文件没关、死锁都可能。

C++ 的解法是 **RAII**（前面 OOP 那篇讲过）：把资源绑到对象的生命周期上。函数因异常退出时，**栈上对象的析构函数照样会被调用**——这叫**栈展开（stack unwinding）**。

```cpp
void foo() {
    std::ifstream file("data.txt");        // 构造时打开
    process(file);                         // 假设这里抛异常
    // 即便不到这一行，file 析构时也会自动关闭
}
```

对比一下用裸资源的版本：

```cpp
void bad() {
    FILE* fp = std::fopen("data.txt", "r");
    process(fp);          // 抛异常的话，fp 永远不会被 fclose
    std::fclose(fp);
}
```

::: callout 异常安全的核心一条
**裸 `new` / `fopen` / `lock` 而没有用 RAII 包起来的代码，遇到异常一定漏。**

办法：
- 内存 → `std::unique_ptr` / `std::shared_ptr` / `std::vector`。
- 文件 → `std::ifstream` / `std::ofstream`。
- 锁 → `std::lock_guard` / `std::unique_lock`。

只要资源都用 RAII 管，**异常安全几乎是免费的**。
:::

## `noexcept`：保证不抛

函数后面加 `noexcept`，表示"这函数承诺不抛异常"：

```cpp
int square(int x) noexcept {
    return x * x;
}
```

它有两个作用：

1. **文档**：告诉调用者"放心，不会抛"。
2. **优化**：编译器知道这函数不会抛，能省掉一些异常处理代码。

如果一个 `noexcept` 函数实际上抛了——程序**直接 `std::terminate`**，不给你机会。所以**加之前要确认**真的不会抛。

入门阶段的建议：

- **析构函数默认 noexcept**（编译器默认就是这样，别打破它）。
- **移动构造 / 移动赋值**：尽量 noexcept，否则 `std::vector` 扩容会偏向调用拷贝。
- **其他函数**：拿不准就别加。

## 异常 vs 错误码：怎么选

异常很方便，但不是万灵药。下面是两条路子的对比：

- **调用者写法**：异常不用每次检查；错误码每次都得 if 一下。
- **性能**：异常抛出时较慢、不抛时几乎免费；错误码性能永远稳定。
- **流程清晰度**：异常让主逻辑干净；错误码会让 if 满天飞。
- **与外部交互**：异常跨语言 / 跨 C API 难传；错误码通用。
- **适合什么**：异常适合"罕见、不该发生的错"；错误码适合"经常会失败的操作"。

**经验：**

- **罕见的、致命的、不在正常控制流里的错** → 用异常（文件读到一半磁盘坏了、内存耗尽、字符串解析失败）。
- **常见的、调用者一定要处理的失败** → 用返回值（"找一个 key，没找到"、"用户输入了非法值"）。

很多 C++ 大型项目（Google、Chrome、LLVM）出于编译开销、嵌入式兼容等考虑，**全程禁用异常**，只用错误码。这是合法的工程选择。

## 不用异常的现代写法

### `std::optional`：可能没有

C++17 引入。表示"一个可能存在、可能不存在的值"：

```cpp
#include <optional>
#include <string>

std::optional<int> parse_int(const std::string& s) {
    try {
        return std::stoi(s);
    } catch (...) {
        return std::nullopt;
    }
}

int main() {
    auto a = parse_int("42");
    auto b = parse_int("oops");

    if (a) std::cout << *a << "\n";        // 42
    if (!b) std::cout << "解析失败\n";

    int x = b.value_or(-1);                 // -1（默认值）
}
```

`optional` 比"用 -1 / 0 / 空字符串当哨兵"清楚得多——**类型本身在告诉你"这玩意可能没有"**。

### `std::expected`（C++23）

`optional` 的进化版——失败时还能带一个**错误信息**：

```cpp
#include <expected>

std::expected<int, std::string> parse_int(const std::string& s) {
    try {
        return std::stoi(s);
    } catch (const std::exception& e) {
        return std::unexpected(e.what());
    }
}

auto r = parse_int("oops");
if (r) std::cout << *r;
else   std::cout << r.error();
```

C++23 才进标准，编译器支持还在跟进。**入门阶段先用 `optional`，看到 `expected` 知道是怎么回事就行**。

## 一个综合例子：配置读取

读一个配置文件，解析每行的 `key=value`，遇到错误能优雅地报告：

```cpp
#include <iostream>
#include <fstream>
#include <optional>
#include <string>
#include <unordered_map>
#include <stdexcept>

class ConfigError : public std::runtime_error {
public:
    ConfigError(const std::string& msg, int line)
        : std::runtime_error("line " + std::to_string(line) + ": " + msg),
          line_(line) {}
    int line() const { return line_; }
private:
    int line_;
};

std::unordered_map<std::string, std::string> load_config(const std::string& path) {
    std::ifstream f(path);
    if (!f) throw ConfigError("cannot open file", 0);

    std::unordered_map<std::string, std::string> cfg;
    std::string line;
    int lineno = 0;
    while (std::getline(f, line)) {
        ++lineno;
        if (line.empty() || line[0] == '#') continue;       // 跳过空行/注释

        auto eq = line.find('=');
        if (eq == std::string::npos) {
            throw ConfigError("missing '='", lineno);
        }

        std::string key   = line.substr(0, eq);
        std::string value = line.substr(eq + 1);
        if (key.empty()) throw ConfigError("empty key", lineno);

        cfg[key] = value;
    }
    return cfg;
}

std::optional<std::string> get(const std::unordered_map<std::string, std::string>& cfg,
                                const std::string& key) {
    auto it = cfg.find(key);
    if (it == cfg.end()) return std::nullopt;
    return it->second;
}

int main() {
    try {
        auto cfg = load_config("app.conf");
        if (auto host = get(cfg, "host")) {
            std::cout << "host = " << *host << "\n";
        } else {
            std::cout << "未配置 host，使用默认 localhost\n";
        }
    } catch (const ConfigError& e) {
        std::cerr << "[ERROR] 配置文件出错：" << e.what() << "\n";
        return 1;
    }
    return 0;
}
```

这段代码示范了**两种风格混用**：

- **加载阶段**——遇到非法格式抛异常。出错就停，错误是"罕见的、致命的"。
- **查询阶段**——返回 `std::optional`。"key 不存在"是**常态**，不该用异常。

工程里这样划分非常常见。

## 常见坑

::: callout 1. 不要用异常做正常控制流
```cpp
try {
    while (true) {
        int x = read();
        if (x < 0) throw "done";   // 拿异常当 break
    }
} catch (...) { }
```
异常**抛出和捕获都有成本**，比 `break` 慢几个数量级。**异常是给"异常情况"的，不是写循环**。
:::

::: callout 2. 析构函数里抛异常
```cpp
~Foo() { throw std::runtime_error("..."); }   // 危险
```
析构函数**默认 noexcept**——抛了直接 `terminate`。即便不 noexcept，**栈展开期间又抛一个新异常**，C++ 也会直接 `terminate`。**析构函数里要么 catch 掉，要么干脆不要做会抛的事**。
:::

::: callout 3. `catch (Exception e)` 而不是 `catch (const Exception& e)`
按值捕获会拷贝（损失派生类信息），还可能出现拷贝异常本身又抛异常的尴尬。**永远写 `catch (const E& e)`**。
:::

::: callout 4. 抛裸字符串、抛 int
```cpp
throw "something wrong";    // 抛 const char*
throw 42;                   // 抛 int
```
能编译能跑——但完全没法被统一处理。**永远抛 `std::exception` 派生的对象**。
:::

::: callout 5. 写 `try { ... } catch (...) { }` 然后什么都不做
"出错了我懒得管"——这是工程里最难调试的代码。错误悄无声息地被吞掉，问题表象出现在十万八千里外。**至少打个日志**：

```cpp
catch (const std::exception& e) {
    std::cerr << "[error] " << e.what() << "\n";
    throw;                  // 上层还要处理就再抛
}
```
:::
